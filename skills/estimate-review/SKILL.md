# Estimate Review Skill

기획/디자인이 확정된 후, 기존 테크 스펙 초안(draft)과 비교하여 **범위 변경 사항을 분석**하고 **일정을 재산정**한다.
"초안에서는 3일이었는데 왜 5일이 되었는지"를 구조화하여 설명할 수 있게 한다.

---

## 1. Input

두 가지를 입력받는다:

### 필수
- **확정된 기획/디자인**: PRD, 기획서, Figma URL, 또는 텍스트 설명
  - "기획서 붙여넣을게", "Figma 링크 줄게", "이렇게 확정됐어" 모두 가능

### 자동 탐색 (없으면 직접 지정)
- **기존 테크 스펙 초안**: `tech-specs/draft-*.md` 파일
  - 프로젝트의 `tech-specs/` 디렉토리에서 관련 초안을 자동으로 찾는다
  - 여러 개 있으면 목록을 보여주고 사용자가 선택한다
  - 직접 경로를 지정할 수도 있다

### 선택
- 참고할 코드 경로 (변경된 코드가 있으면)

---

## 2. 생성 프로세스

### Step 1: 기존 초안 로드

1. `tech-specs/` 디렉토리에서 `draft-*.md` 파일 목록을 검색한다
2. 기능명이나 날짜로 관련 초안을 매칭한다
3. 초안의 가정, 범위, 공수 추정을 추출한다

### Step 2: 확정 기획 분석

1. 확정된 기획/디자인의 요구사항을 파악한다
2. 초안의 가정(Assumptions)과 대조하여 맞았는지/틀렸는지 판별한다
3. 범위(Scope)의 포함/제외 항목과 대조하여 변경 사항을 식별한다

### Step 3: 변경 사항 비교

아래 관점에서 초안과 확정본을 비교한다:

- **가정 검증**: 초안의 가정이 맞았는지, 틀렸다면 어떻게 달라졌는지
- **범위 변경**: 추가된 기능, 제거된 기능, 변경된 기능
- **복잡도 변화**: 예상보다 단순해진 것 / 복잡해진 것
- **의존성 변화**: 새로 생긴 외부 의존성 또는 해소된 의존성
- **리스크 변화**: 해소된 리스크, 새로 발견된 리스크

### Step 4: 일정 재산정

- 초안의 공수 추정 테이블을 기반으로 변경된 항목만 업데이트
- 추가/삭제된 작업은 별도 행으로 명시
- 변경 이유를 비고 컬럼에 기록
- 최종 합계와 초안 대비 변동폭을 표시

### Step 5: 저장

- 프로젝트 `tech-specs/` 디렉토리에 저장
- 파일명: `review-{기능명-kebab-case}-{YYYY-MM-DD}.md`
- 대화창에 먼저 보여주고, 사용자 확인 후 저장

## 3. 일정 재산정 포맷

```markdown
# [기능명] 일정 재산정

**작성자:** 이용진
**작성일:** {오늘 날짜 YYYY-MM-DD}
**기존 초안:** `{초안 파일명}`
**상태:** Review

---

## 요약

> 초안 대비 변경된 핵심 사항을 2~3줄로 요약.
> 총 공수: N일 → M일 (±Xd)

---

## 가정 검증

| 초안 가정 | 결과 | 영향 |
|-----------|------|------|
| 가정 1: ○○라고 가정 | ✅ 맞음 | 변경 없음 |
| 가정 2: △△라고 가정 | ❌ 틀림 — 실제로는 □□ | 작업 추가 필요 |
| 가정 3: ◇◇라고 가정 | ⚠️ 부분적 — ◎◎로 변경 | 공수 증가 |

---

## 범위 변경

### 추가된 항목

| 항목 | 설명 | 예상 공수 |
|------|------|-----------|
| 추가 기능 1 | 기획서에서 새로 요구 | +1d |

### 제거된 항목

| 항목 | 설명 | 감소 공수 |
|------|------|-----------|
| 제거 기능 1 | 다음 단계로 이관 | -0.5d |

### 변경된 항목

| 항목 | 초안 | 확정 | 공수 변동 |
|------|------|------|-----------|
| 기능 A | 단순 목록 | 필터/정렬 포함 | +1d |

---

## 공수 재산정

| 작업 | 영역 | 초안 | 재산정 | 변동 | 변경 이유 |
|------|------|------|--------|------|-----------|
| 작업 1 | FE | 0.5d | 0.5d | - | 변경 없음 |
| 작업 2 | BE | 1d | 1.5d | +0.5d | 기획서에서 요구사항 추가 |
| 작업 3 (신규) | FE | - | 1d | +1d | 초안에 없던 기능 추가 |
| 작업 4 (삭제) | BE | 0.5d | - | -0.5d | 다음 단계로 이관 |
| QA / 버그 수정 | 공통 | 1d | 1.5d | +0.5d | 범위 증가에 따른 조정 |
| **합계** | | **3d** | **5d** | **+2d** | |

> 영역: FE(프론트엔드), BE(백엔드), Infra(인프라), 공통

## 일정 변동 근거 요약

> 윗분에게 보고할 때 사용할 수 있는 요약.

**초안 대비 +Xd 증가 원인:**
1. ○○ 기능 추가 (+Nd): 기획서에서 ~~가 요구됨
2. △△ 복잡도 증가 (+Nd): 단순 목록 → 필터/정렬 포함
3. □□ 감소 (-Nd): 다음 단계로 이관

**결론:** 기획 확정에 따라 총 공수 N일 → M일로 조정 필요.
```

## 4. 섹션별 작성 가이드

### 요약
- 공수 변동폭을 반드시 포함 (N일 → M일)
- 한눈에 "늘었는지 줄었는지" 파악 가능하게

### 가정 검증
- 초안의 가정을 하나씩 검증. ✅/❌/⚠️ 아이콘으로 직관적 표시
- 틀린 가정이 공수에 미치는 영향을 반드시 연결

### 범위 변경
- 추가/제거/변경을 명확히 구분
- 각 항목에 예상 공수 영향을 함께 기재
- "왜 추가되었는지" (기획서 요구 vs 기술적 필요)를 구분

### 공수 재산정
- 초안 테이블을 그대로 가져오고 옆에 재산정 컬럼 추가
- 변동이 없는 항목도 포함 (전체 그림 유지)
- 변경 이유 컬럼이 핵심 — 이게 보고 자료가 됨

### 일정 변동 근거 요약
- **가장 중요한 섹션**. 윗분에게 "왜 일정이 늘었는지" 설명할 때 이 섹션만 보여주면 됨
- 증가/감소 원인을 항목별로 나열
- 감정이 아닌 사실 기반으로 작성 ("복잡해서"가 아니라 "필터/정렬 UI가 추가되어")

## 5. Notion 저장

Notion 저장이 요청되면 기존 `tech-spec` 스킬의 `references/notion-agent.md`를 참고한다.

저장 시 기본 설정:
- 페이지 타이틀: `[Review] {기능명} - YYYY-MM-DD`
- 상태 속성: `Review`
- 태그: `estimate-review`
