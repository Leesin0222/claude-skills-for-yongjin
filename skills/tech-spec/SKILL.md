# Tech Spec Skill

뱅크샐러드 테크 스펙 포맷을 기반으로, 다양한 Input으로부터 테크 스펙 초안을 생성한다.

---

## 1. Input 타입 분류

세 가지 Input을 각각 또는 조합하여 처리한다.

| 타입 | 설명 | 처리 방향 |
|------|------|-----------|
| **텍스트 설명** | 구현할 기능을 자연어로 설명 | 의도를 파악해 모든 섹션 추론 생성 |
| **PRD / 기획서** | PM이 작성한 요구사항 문서 | 비즈니스 목표 → 기술 구현으로 번역 |
| **코드** | 기존 코드베이스 또는 변경 diff | 코드 분석 → 역방향으로 스펙 재구성 |

Input이 여러 개 있으면 우선순위: 코드 > PRD > 텍스트 순으로 구체성을 보완한다.

---

## 2. 생성 프로세스

### Step 1: Input 분석

Input 타입을 식별하고 핵심 정보를 추출한다.

**텍스트 설명인 경우:**
- 기능명, 대상 사용자, 핵심 동작을 파악
- 불명확한 부분은 합리적으로 가정하고, 가정한 내용은 문서에 명시

**PRD인 경우:**
- 비즈니스 목표, 기능 요구사항, 제약조건 추출
- 기술적으로 번역이 필요한 부분 식별 (예: "빠르게" → 구체적 성능 목표)
- 기획에서 누락된 엣지 케이스 명시

**코드인 경우:**
- 주요 컴포넌트, 데이터 흐름, 의존성 파악
- 코드에서 의도를 역추론 (왜 이렇게 짰는지)
- 개선 가능 지점이나 기술 부채 있으면 Other Considerations에 기록

### Step 2: 섹션별 생성

아래 **테크 스펙 포맷**을 순서대로 채운다. 정보가 불충분한 섹션은 `[작성 필요]` 태그와 함께 가이드를 남긴다.

### Step 3: 출력

- 기본: 마크다운 포맷으로 대화창에 출력
- Notion 저장 요청 시: `references/notion-agent.md` 참고

---

## 3. 테크 스펙 포맷 (뱅크샐러드 기반)

```markdown
# [기능명] 테크 스펙

**작성자:** [이름]  
**작성일:** YYYY-MM-DD  
**상태:** Draft / In Review / Approved

---

## 요약 (Summary)

> 이 테크 스펙을 3줄 이내로 요약한다. 누가/무엇을/언제/어디서/왜를 명확하게.

[예시: Bottom Navigation 영역을 유저가 원하는 순서로 커스텀할 수 있게 한다.
서버 API 없이 순서를 로컬에 저장하고 불러온다.]

---

## 배경 (Background)

> 이 기능을 왜 만드는가. 동기, 사용자 문제, 이전 시도 여부.

- 어떤 사용자 문제를 해결하는가?
- 이 기능이 없으면 어떤 일이 발생하는가?
- 이전에 유사한 시도가 있었는가?

---

## 목표 (Goals)

> 예상 결과를 Bullet Point로 나열. 이 목표로 프로젝트 성공 여부를 평가한다.

- [ ] 목표 1
- [ ] 목표 2
- [ ] 목표 3

---

## 목표가 아닌 것 (Non-Goals)

> 의도적으로 하지 않거나 이번 범위에서 제외되는 것. 범위 폭주를 막는 섹션.

- 이번 작업에 포함되지 않는 기능
- 의도적으로 제외하는 항목과 그 이유

---

## 계획 (Plan)

> 가장 긴 파트. 기술적/엔지니어링적 접근 방법을 상세히 기술.
> 확실히 결정되지 않은 부분은 고려 중인 옵션 목록으로 남긴다.

### 기술 설계

[시스템 구조, 컴포넌트 간 상호작용 설명]

### 데이터 흐름

[시퀀스 다이어그램, 데이터 흐름 설명 — 필요시 Mermaid 다이어그램 포함]

```mermaid
sequenceDiagram
    ...
```

### 핵심 로직

[의사 코드(Pseudo-code) 또는 핵심 알고리즘 설명]

```typescript
// 핵심 로직 의사 코드
```

### API / 인터페이스

[HTTP 엔드포인트, 요청/응답 포맷, 에러 명세 — 해당하는 경우]

---

## 이외 고려 사항 (Other Considerations)

> 검토했으나 하지 않기로 결정된 사항들. 이미 논의된 주제가 반복되지 않도록.

| 고려 사항 | 결론 | 이유 |
|-----------|------|------|
| 대안 A | 채택 안 함 | 이유 |
| 대안 B | 채택 안 함 | 이유 |

---

## 마일스톤 (Milestones)

> 테크 스펙 기반으로 추정한 작업 일정. 실험 계획, 배포 날짜 포함.

| 기간 | 작업 |
|------|------|
| ~ MM/DD | 작업 1 |
| MM/DD ~ MM/DD | 작업 2 |
| MM/DD | 코드 프리즈 / QA |
| MM/DD | 배포 |

---

## 관련 문서

- 기획서: [링크]
- 디자인 시안: [링크]
- 관련 PR: [링크]
- 참고 테크 스펙: [링크]
```

---

## 4. 섹션별 작성 가이드

### 요약
- 3줄 이내. 길면 배경으로 이동.
- 기술 용어보다 "무엇을 왜" 중심으로.

### 배경
- PRD가 있으면 비즈니스 컨텍스트를 직접 인용하지 않고 개발자 관점으로 재해석.
- 코드가 있으면 기존 구조의 한계점을 서술.

### 목표 vs 목표가 아닌 것
- Goals는 측정 가능하게. "빠르게" → "p99 응답시간 200ms 이하"
- Non-Goals는 구체적일수록 좋다. 범위 폭주 방지가 핵심.

### 계획
- 불확실한 부분은 숨기지 말고 옵션으로 열거.
- 복잡한 로직은 반드시 의사 코드로 먼저 표현.
- 다른 시스템과 연동되면 시퀀스 다이어그램 포함.

### 마일스톤
- 태스크를 컴포넌트/로직 단위로 쪼갠 후 공수 산정.
- 프론트/백엔드/디자인/QA 일정 통합 관리.
- `[작성 필요]` 남기는 것보다 대략적 추정이라도 적는 게 낫다.

---

## 5. Input별 특이사항

### 코드 Input 처리 시 추가 분석
코드베이스가 Input으로 들어오면 Plan 섹션에 아래를 추가한다:

```markdown
### 기존 코드 분석

**주요 컴포넌트:**
- `ComponentName`: 역할 설명

**데이터 흐름:**
- 현재 흐름 설명

**기술 부채 / 개선 포인트:**
- 발견된 이슈 (Other Considerations로 이동 가능)
```

### 텍스트 설명만 있는 경우
가정(Assumption)을 명시하는 섹션을 Summary 아래 추가한다:

```markdown
## 가정 (Assumptions)

> Input이 불충분해 아래와 같이 가정했습니다. 검토 후 수정해주세요.

- 가정 1: ...
- 가정 2: ...
```

Notion 저장이 요청되면 `references/notion-agent.md`를 읽고 진행한다.

저장 시 기본 설정:
- 페이지 타이틀: `[Tech Spec] {기능명} - YYYY-MM-DD`
- 상태 속성: `Draft`
- 태그: `tech-spec`
