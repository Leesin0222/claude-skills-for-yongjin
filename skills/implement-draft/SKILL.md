# Implement Draft Skill

기능 설명을 자연어로 받아, 현재 프로젝트의 코드베이스를 분석한 뒤 **기존 패턴에 맞는 코드 초안**을 생성한다.
기획서/디자인이 없는 초기 단계에서도 사용 가능하며, 생성된 코드는 "초안"으로서 사용자 리뷰 후 수정하는 것을 전제한다.

---

## 1. Input

기능 설명 텍스트를 주 입력으로 받는다.

- 자연어 기능 설명 ("사용자 프로필 페이지 만들어줘")
- 슬랙/회의에서 나온 기능 아이디어
- 참고할 코드 경로 (선택 — 없으면 자동 탐색)

---

## 2. 생성 프로세스

### Step 1: 코드베이스 분석

아래 순서로 현재 프로젝트를 파악한다.

1. **프로젝트 설정 읽기**
   - `.claude/CLAUDE.md`에서 기술 스택, 주요 코드 경로 정보 확인
   - `package.json`, `pyproject.toml`, `go.mod`, `Cargo.toml` 등 프로젝트 메타 파일로 스택 보완

2. **디렉토리 구조 파악**
   - 프로젝트 루트의 주요 디렉토리 구조를 탐색 (Glob)
   - 소스 코드 디렉토리의 패턴 파악 (예: `src/features/`, `src/components/`, `app/`, `pages/`)

3. **관련 코드 탐색**
   - 요청된 기능과 관련된 기존 코드를 Grep/Glob으로 검색
   - 유사한 기능이 이미 있으면 해당 코드의 패턴을 참고 대상으로 삼는다

4. **아키텍처 패턴 추출**
   - 기존 코드에서 반복되는 패턴 파악:
     - 디렉토리 구조 컨벤션 (feature-based, layer-based 등)
     - 파일 네이밍 컨벤션 (camelCase, kebab-case, PascalCase)
     - import/export 패턴
     - 상태 관리 방식
     - API 호출 패턴
     - 에러 처리 패턴
     - 타입 정의 방식

### Step 2: 구현 계획 제시

코드 작성 전에 반드시 사용자에게 계획을 보여주고 확인을 받는다.

```markdown
## 구현 계획

### 파악한 프로젝트 구조
- 기술 스택: {파악한 스택}
- 아키텍처 패턴: {파악한 패턴}
- 참고한 기존 코드: {경로 목록}

### 생성할 파일
| 파일 경로 | 설명 | 신규/수정 |
|-----------|------|-----------|
| `src/features/xxx/index.ts` | 메인 로직 | 신규 |
| `src/features/xxx/types.ts` | 타입 정의 | 신규 |
| `src/routes/xxx.ts` | 라우트 등록 | 수정 |

### 수정할 기존 파일
| 파일 경로 | 변경 내용 |
|-----------|-----------|
| `src/routes/index.ts` | 새 라우트 import 추가 |

### 가정 사항
- 가정 1: ...
- 가정 2: ...

> 이 계획대로 진행할까요? 수정이 필요하면 말씀해주세요.
```

**사용자가 확인하면** Step 3으로 진행한다.
**수정 요청이 오면** 계획을 조정한 뒤 다시 제시한다.

### Step 3: 코드 생성

확인된 계획에 따라 코드를 작성한다.

**코드 작성 원칙:**

1. **기존 패턴 준수**
   - 프로젝트에서 파악한 네이밍, 구조, 스타일을 그대로 따른다
   - 프로젝트에 없는 새로운 패턴이나 라이브러리를 도입하지 않는다
   - 기존 코드에서 사용하는 유틸리티/헬퍼가 있으면 그것을 재사용한다

2. **최소 동작 코드**
   - 핵심 기능이 동작하는 최소한의 코드를 작성한다
   - 에러 처리, 로깅, 최적화는 기본적인 수준만 포함한다
   - TODO 주석으로 추후 보완이 필요한 부분을 명시한다

3. **안전한 수정**
   - 기존 파일 수정 시 Edit 도구를 사용하여 최소 범위만 변경한다
   - 기존 로직을 삭제하거나 변경하지 않고, 새 코드를 추가하는 방향으로 작성한다
   - 기존 테스트가 깨지지 않도록 주의한다

4. **명확한 초안 표시**
   - 파일 상단에 `// [DRAFT] implement-draft로 생성된 초안입니다. 리뷰 후 사용하세요.` 주석을 추가한다
   - 확신이 없는 부분에는 `// TODO: [implement-draft]` 주석을 남긴다

### Step 4: 결과 요약

코드 생성 완료 후 아래 포맷으로 결과를 요약한다.

```markdown
## 구현 완료 요약

### 생성/수정된 파일
| 파일 | 상태 | 설명 |
|------|------|------|
| `path/to/file` | 신규 | 설명 |
| `path/to/file` | 수정 | 변경 내용 |

### TODO 목록
- [ ] `path/to/file:L42` — 에러 처리 보완 필요
- [ ] `path/to/file:L78` — 비즈니스 로직 확인 필요

### 다음 단계 제안
- 리뷰 후 TODO 항목 처리
- 테스트 코드 작성
- 기획/디자인 확정 후 UI 보완
```

## 3. 코드베이스 분석 가이드

### 기술 스택별 탐색 전략

**JavaScript/TypeScript 프로젝트:**
- `package.json` → 프레임워크 및 주요 의존성 파악
- `tsconfig.json` → TypeScript 설정 (paths alias 등)
- `src/` 또는 `app/` 하위 구조 → 아키텍처 패턴
- 기존 컴포넌트/모듈 1~2개를 샘플로 읽어 코딩 스타일 파악

**Python 프로젝트:**
- `pyproject.toml` 또는 `requirements.txt` → 프레임워크 파악
- `src/` 또는 프로젝트 루트 하위 구조
- `__init__.py` 패턴으로 모듈 구조 파악

**Go 프로젝트:**
- `go.mod` → 모듈명 및 의존성
- `cmd/`, `internal/`, `pkg/` 구조 파악
- 기존 핸들러/서비스 패턴 참고

**기타:**
- 프로젝트 루트의 빌드/설정 파일로 스택 추론
- `src/` 하위 디렉토리 구조로 아키텍처 유추

### 패턴 파악 체크리스트

기존 코드에서 아래 항목을 확인한다:

- [ ] 디렉토리 구조 컨벤션 (feature-based / layer-based / hybrid)
- [ ] 파일 네이밍 (camelCase / kebab-case / PascalCase / snake_case)
- [ ] export 방식 (named export / default export / barrel export)
- [ ] 상태 관리 (전역 store / context / props drilling)
- [ ] API 호출 패턴 (fetch wrapper / axios instance / tRPC 등)
- [ ] 에러 처리 방식 (try-catch / Result type / error boundary)
- [ ] 타입 정의 위치 (같은 파일 / types.ts / *.d.ts)
- [ ] 테스트 파일 위치 (__tests__ / *.test.ts / *.spec.ts)

---

## 4. 안전장치

### 기존 코드 보호

- 기존 파일 수정 시 **추가(append/insert)** 위주로 작업하고, 기존 코드 삭제/변경은 최소화
- 수정할 파일은 반드시 먼저 Read로 전체 내용을 파악한 뒤 Edit으로 변경
- 한 파일에 대량 변경이 필요하면 사용자에게 별도 확인

### 초안 코드 표시

모든 생성 코드에 초안임을 명시하여 프로덕션 코드와 구분:

```typescript
// [DRAFT] implement-draft로 생성된 초안입니다. 리뷰 후 사용하세요.
// 생성일: {YYYY-MM-DD}
```

### 제한 사항

- 환경 변수(.env), 시크릿, 인증 정보는 생성하지 않는다
- DB 마이그레이션 파일은 직접 생성하지 않고, 필요 시 TODO로 안내한다
- 기존 테스트 파일은 수정하지 않는다 (새 테스트 파일 생성은 가능)

---

## 5. 슬래시 커맨드 플로우

`/implement-draft` 호출 시:

1. 프로젝트 `.claude/CLAUDE.md`에서 코드 구조 정보를 읽는다.
2. 사용자에게 요구사항을 입력받는다:
   - 기능명
   - 기능 설명 (자연어 — "이런 기능 만들어줘")
   - 참고할 코드 경로 (선택 — 없으면 자동 탐색)
3. 코드베이스를 분석하고 구현 계획을 제시한다.
4. 사용자 확인 후 코드를 생성/수정한다.
5. 결과 요약을 출력한다.
