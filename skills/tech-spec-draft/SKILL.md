# Tech Spec Draft Skill

기능 아이디어 또는 대략적인 기능 설명을 받아, **일정 산정에 필요한 수준**의 테크 스펙 초안을 생성한다.
기획서/디자인이 없는 초기 단계에서 사용한다.

---

## 1. Input

기능 설명 텍스트를 주 입력으로 받는다.

- 윗분이 전달한 기능 아이디어
- 대략적인 기능 요구사항
- 슬랙/회의에서 나온 기능 설명

참고할 코드 경로가 있으면 추가로 받되, 없어도 프로젝트 코드베이스를 탐색하여 관련 코드를 파악한다.

---

## 2. 생성 프로세스

### Step 1: 기능 이해

- 기능의 핵심 동작을 파악한다
- 불명확한 부분은 합리적으로 가정하고, 가정 목록을 문서에 포함한다
- 관련 코드가 있으면 기존 구조를 파악하여 구현 난이도를 추정한다

### Step 2: 초안 생성

아래 **테크 스펙 초안 포맷**을 채운다. 모든 섹션을 가볍게 채우는 것이 목표다.

### Step 3: 저장

- 프로젝트 루트의 `tech-specs/` 디렉토리에 마크다운 파일로 저장한다
- 파일명: `draft-{기능명-kebab-case}-{YYYY-MM-DD}.md`
- 디렉토리가 없으면 생성한다
- Notion 저장이 추가로 요청되면 `references/notion-agent.md`를 참고한다

---

## 3. 테크 스펙 초안 포맷

```markdown
# [기능명] 테크 스펙 초안

**작성자:** 이용진
**작성일:** {오늘 날짜 YYYY-MM-DD}
**상태:** Draft (일정 산정용)

---

## 요약

> 이 기능이 무엇인지 2~3줄로 설명한다.

---

## 가정 (Assumptions)

> 기획서/디자인이 없으므로 아래와 같이 가정했습니다. 검토 후 수정해주세요.

- 가정 1: ...
- 가정 2: ...

---

## 범위 (Scope)

### 포함

- 이번에 개발할 것들

### 제외

- 이번에 하지 않을 것들과 이유

---

## 구현 방향

> 어떻게 만들지 대략적인 접근 방법. 확정이 아닌 방향성.

### 주요 작업 항목

- [ ] 작업 1 — 간단 설명
- [ ] 작업 2 — 간단 설명
- [ ] 작업 3 — 간단 설명

### 기존 코드 영향도

- 수정이 필요한 기존 코드/모듈
- 새로 만들어야 하는 것

---

## 리스크 및 미확인 사항

| 항목 | 내용 | 영향도 |
|------|------|--------|
| 리스크 1 | 설명 | 높음/중간/낮음 |
| 미확인 사항 1 | 기획 확정 시 확인 필요 | - |

---

## 공수 추정

| 작업 | 영역 | 추정 기간 | 비고 |
|------|------|-----------|------|
| 작업 1 | FE | 0.5d | |
| 작업 2 | BE | 1d | |
| 작업 3 | FE/BE | 0.5d | |
| QA / 버그 수정 | 공통 | 1d | |
| **합계** | | **Nd** | |

> 영역: FE(프론트엔드), BE(백엔드), Infra(인프라), 공통
> ⚠️ 기획/디자인 확정 후 변동 가능

---

## 의존성

### 외부 의존성 (다른 팀/시스템)

| 의존 대상 | 필요한 것 | 영향도 | 상태 |
|-----------|-----------|--------|------|
| 예: 서버팀 | API 엔드포인트 제공 | 높음 | 미요청 |

### 기존 일정 영향도

- 현재 진행 중인 작업에 미치는 영향
- 병렬 진행 가능 여부

---

## 기획 확정 시 필요한 결정 사항

- [ ] 결정 필요 1: ...
- [ ] 결정 필요 2: ...

---

## 관련 코드

- `경로`: 설명
```

## 4. 섹션별 작성 가이드

### 요약
- 기술 용어 최소화. "무엇을 왜" 중심.
- 윗분이 읽어도 이해할 수 있는 수준.

### 가정
- 기획서가 없으므로 반드시 포함. 추후 검증 대상.
- "~라고 가정합니다" 형태로 명확히.

### 범위
- 제외 항목이 특히 중요. 범위 폭주 방지.
- "이건 다음 단계에서" 같은 판단을 미리 드러낸다.

### 구현 방향
- 확정이 아닌 방향성임을 명시.
- 기존 코드가 있으면 어디를 건드려야 하는지 파악.
- 상세 설계(API 명세, 다이어그램 등)는 이 단계에서 불필요.

### 리스크 및 미확인 사항
- 일정에 영향을 줄 수 있는 불확실성.
- 기획 확정 시 답이 나올 항목 구분.

### 공수 추정
- 개발자 1명 기준, 일 단위(d)로 추정.
- 보수적으로 산정. 버퍼 포함.
- QA/버그 수정 시간 별도 항목으로.
- FE/BE/Infra 영역을 반드시 구분. 인력 배분 판단의 근거.

### 의존성
- 다른 팀 작업이 필요하면 반드시 명시. 이게 빠지면 일정이 틀어질 수 있음.
- 현재 진행 중인 작업과 충돌 여부도 확인.

### 기획 확정 시 필요한 결정 사항
- 기획자/디자이너에게 전달할 질문 목록.
- 이 결정에 따라 공수가 달라질 수 있는 항목.

---

## 5. Notion 저장

Notion 저장이 요청되면 기존 `tech-spec` 스킬의 `references/notion-agent.md`를 참고한다.

저장 시 기본 설정:
- 페이지 타이틀: `[Draft] {기능명} - YYYY-MM-DD`
- 상태 속성: `Draft`
- 태그: `tech-spec-draft`
